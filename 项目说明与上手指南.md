## 一、一句话概括

**多智能体（大模型）在离线阶段不断"发现规律、积累经验"，把它们编译成一个
规则库 + 案例库；车辆真正运行时只做毫秒级的库匹配，全程不调用任何大模型、
不联网。**

这样既拿到了大模型的推理与泛化能力，又彻底回避了大模型"单次调用要好几秒"
与极限避撞"留给决策的时间往往不到 0.3 秒"之间的根本矛盾。

## 二、要解决的问题与核心思路

### 2.1 背景

极限工况避撞（例如侧向高速逼近的卡车、突然横穿车道的障碍物）要求在极短
时间内从多种机动里选出"伤害最小"的那个 —— 包括全力制动、急变道、变道+制动，
乃至 T 形漂移（把车尾这种吸能结构转向撞击点，保护乘员舱与动力电池）。

前作 SACA（arXiv:2504.00115）用"一次大模型调用"来做这个仲裁。问题是：
大模型一次推理要 1~10 秒（即便用最轻量模型、关闭思考也要 10 秒上下才能
完成一轮多智能体协商），而留给避撞决策的时间窗常常不足 300 毫秒。
**大模型再聪明，也来不及。**

本项目历经三个架构阶段的演进：

| 阶段 | 架构 | 触发时刻时延 | 问题 |
|---|---|---|---|
| v0（SACA legacy） | 单次 LLM 调用 | 秒级 | 完全来不及 |
| v1（预谋式多智能体） | 风险爬升期后台深思、预存策略树、触发时查表 | 查表 ms 级，但深思 12~135 s 常赶不上预谋窗口 | 首遇场景仍不安全 |
| **v2（当前，经验编译）** | **LLM 全部离线，运行时只匹配编译产物** | **1~2 ms，恒定** | 库未覆盖时退化为最小风险机动（并自动触发离线补齐） |

### 2.2 核心思路：把"思考"和"执行"彻底分离到两个平面

| | 在线平面（Reflex / 反射） | 离线平面（Factory / 工厂） |
|---|---|---|
| 何时运行 | 车辆实时运行时 | 事先/事后，不限时 |
| 用不用大模型 | **绝不**（代码层面 runtime/ 不 import openai） | 大模型只在这里 |
| 做什么 | 规则/案例库匹配，选机动 | 发现规律、积累案例、丰富库 |
| 时间尺度 | **毫秒级（实测全场景 1~2 ms）** | 秒~分钟级，无所谓 |
| 网络依赖 | 无（可断网运行） | 需要 OpenAI API |
| 类比 | 运行"编译好的程序" | "编译器"在编译 |

一句话：**多智能体是策略编译器，运行时执行的是编译产物。**

### 2.3 为什么"案例再多也不影响实时"

这是本架构最容易被质疑的点，答案是：**查询编码本地化**。

案例检索的传统瓶颈不在案例数量，而在"每次查询要调一次大模型的 embedding
接口"（约 0.3~1 秒网络往返）。本项目把场景直接编码成一个 **32 维本地数值
特征向量**（TTC、方位扇区、横向速度、走廊占用……全部由场景几何本地算出，
零网络调用），于是：

- 规则匹配：按守卫条件逐条过滤，实测 ~0.01 ms；
- 案例 k-NN：一次 numpy 矩阵-向量乘算出全库余弦相似度，
  10³ 案例 <1 ms，10⁵ 案例约 1~10 ms（如需更大规模可换 HNSW 索引，<1 ms）；
- 结论：**库规模增长不构成实时性威胁 —— 实时性由"构造方式"保证，
  而非靠限制库大小。**

### 2.4 库没覆盖怎么办？——安全兜底 + 自动补齐

在线遇到库没覆盖的场景时：立即执行**最小风险机动**（code 0 全力制动，
控制权永不悬空），同时把该场景（完整快照 + 特征）写入反馈队列；下一个
工厂批次会优先消化队列，把它变成案例 —— 同一场景第二次出现时就有解了。
规则与案例给出不同结论时同理（见"冲突驱动自愈"）。

### 3.1 实测时延预算（在线触发路径，热态）

| 环节 | 模块 | 实测耗时 |
|---|---|---|
| ① 反射层风险监视 | monitor.py | ~1 ms（含 HJ 网络推理；冷启动首跑 ~15 ms 为 torch 初始化） |
| ② 特征化（含制动仿真接地） | features.py | ~0.3 ms |
| ③ 规则匹配 | rule_engine.py | ~0.01 ms |
| ③ 案例 k-NN | case_index.py | ~0.05–0.5 ms（当前库规模） |
| ④ 演化分类 + 选支 | policy_match.py | ~0.1 ms |
| ⑤ 保守仲裁（仅分歧时） | arbiter.py | ~0.1 ms（两次机动仿真） |
| **触发路径合计** | executor.py | **约 1–2 ms**（全程零 LLM、零网络） |

结局仿真与日志落盘不在关键路径上，不计入。数字可在任意运行记录的
`timings` 字段复核（`--json` 或 `Results/runs/*.json`）。

## 四、创新点

1. **双平面经验编译**：把大模型时延彻底摊销到离线，在线是零思考的编译产物。
   相比 DiLu / LeapAD / SafeDrive 等"慢系统仍在运行时路径上"的方案，本架构
   在线完全无 LLM，实时性由构造保证而非尽力而为。

2. **网格扫描式规律发现**：大模型负责"设计实验 + 归纳规律"，确定性仿真器提供
   "物理真值"，归纳出的规则边界再逐点回到网格做**数值交叉验证** ——
   大模型的幻觉边界会被数值复核当场拦下并打回重提。这是"让大模型参与但
   不轻信大模型"的关键机制。

3. **规则/案例的原理性分工**：规则覆盖"某机动严重度明显占优的有规律区域"，
   案例覆盖"严重度接近、或涉及价值权衡（如是否伤及行人）的模糊区域"。
   两者共用同一套特征语言，可无限扩展（新特征无需改引擎）。

4. **覆盖门控激活**：每一分大模型 token 都以"库尚未正确处理该场景"为前提 ——
   已覆盖的队列项跳过、已被规则覆盖的规律区域跳过归纳、模糊格点逐个查覆盖、
   已裁决的冲突不再重复入队。特别地，**"规则命中但严重度次优"不算覆盖**，
   而是作为**反例**回喂归纳器用于修订规则。

5. **冲突驱动的自愈学习**：规则与案例分歧时，在线做确定性保守仲裁（两个候选
   各仿真取预测伤害小者，~0.1 ms），冲突入队；离线辩论智能体（低思考、少步数）
   裁决后，败诉方被处理（案例退役 / 规则待修订），同一分歧不再复现。
   这也是"不完美的领域种子规则被证据自动纠正"的通道。

6. **规则全生命周期 + 抗漂移**：规则有 candidate→active→deprecated 生命周期，
   携带来源（**可复实例化**的场景族定义）与验证统计；每批次抽样复验现役规则，
   仿真器或特征语义演进导致规则失效时自动废弃（本项目开发中真实发生过一次：
   特征语义升级后一条旧规则匹配域消失，复验自动将其废弃）。

7. **HJ 可达性的正确定位**：HJ 可达性值网络本质是"对状态的风险评估依据"，
   本项目用作 (1) 反射层风险监视信号、(2) 机动验证时的末态安全约束参考
   （末态 HJ 风险不得劣于"不干预"基线），而非最终裁决者 —— 最终裁决用
   运动学仿真严重度。HJ 权重取自 `HJ_Reachability/safe_value_params.pth`
   （数值逐位可复现）。

8. **慢教快的蒸馏出口**：每次委员会深思都自动导出为一条微调样本
   （`fine-tuning/maca_distill.jsonl`，标准 OpenAI messages 微调格式；
   该文件按需自动创建），为将来把离线经验蒸馏进车端小模型留出接口。

## 五、机动目录与决策码

决策码 0~7 的语义与已发表的 SACA 框架完全一致（研究贡献保持不变）：

| 码 | 机动 | 运动学剖面 | 典型适用 |
|---|---|---|---|
| 0 | 全力制动（AEB） | 纵向 −8 m/s² 直行至停 | 制动距离足够 / 万能兜底 |
| 1 | 急向左变道 | 侧向 bang-bang ±3 m/s²，4 m 到位侧速归零；纵向匀速 | 左侧走廊空、需保速逃逸 |
| 2 | 急向右变道 | 同上镜像 | 右侧走廊空、需保速逃逸 |
| 3 | 向左变道 + 制动 | 侧向同 1，叠加纵向 −4 m/s² | 左侧空且需要降速 |
| 4 | 向右变道 + 制动 | 同上镜像 | 右侧空且需要降速 |
| 5 | T 形漂移（车头向左） | 两段式：①旋转段(~0.9s)纵向 −5 + 侧向 −5 甩尾把航向转至垂直；②滑行段摩擦沿速度方向擦除动能至停 | 碰撞不可避、威胁从**右**来 → 车尾朝右受撞 |
| 6 | T 形漂移（车头向右） | 同上镜像 | 碰撞不可避、威胁从**左**来 → 车尾朝左受撞 |
| 7 | 不干预 | 匀速 | 低风险 / 威胁已让行 |

T 形漂移的领域依据：车辆吸能结构在前后端，侧面结构弱且 EV 电池防护差 ——
碰撞不可避免时应让车尾（或车头）而非侧面承受撞击。触发前提（领域知识）：
横向逼近目标 TTC ≤ 1.3 s、位于前方、横距 < 8 m、横向逼近速度 > 4 m/s。

**应急策略树（policy tree）**：规则与案例的载荷不是单个机动，而是一棵
条件分支树 —— 每个分支由封闭集合的守卫条件标注：

| 分支守卫 | 含义 |
|---|---|
| `primary_threat_maintains` | 主威胁保持当前运动（最常命中） |
| `primary_threat_yields` | 主威胁减速/让行/脱离碰撞路径 |
| `primary_threat_accelerates` | 主威胁比预期更快逼近 |
| `secondary_threat_activates` | 主威胁易主（另一目标成为决定性约束） |
| `default` | 兜底分支（必须存在） |

触发时刻由 `policy_match.classify_evolution` 对比预览帧与触发帧，纯运动学地
把"场景实际怎么演化了"分类到上述集合，然后毫秒级选中分支执行 ——
不解析任何自由文本。

## 六、特征语言（双平面的共同词汇表）

离线工厂在这套特征上归纳规则守卫；在线平面微秒级求值它们。全部由
`runtime/features.py::extract_features` 本地计算。

### 离散特征（8 项）

| 特征 | 取值域 | 含义 |
|---|---|---|
| `road_topology` | Normal Road / Intersection / Roundabout | 道路拓扑 |
| `threat_kind` | truck / suv / car / pedestrian / obstacle / other | 主威胁类型桶 |
| `approach_sector` | front / front_right / right / rear_right / rear / rear_left / left / front_left | 主威胁方位（8 向扇区，每 45°） |
| `pedestrian_side` | none / left / right | 行人所在侧（决定变道/漂移禁行方向） |
| `corridor_left_free` | true / false | 左走廊（x∈[-5,25], y∈[-5,-1]）是否可变道 |
| `corridor_right_free` | true / false | 右走廊镜像 |
| `braking_sufficient` | true / false | **仿真接地**：直接推演 code 0，severity ≤1 才为 true |
| `on_collision_course` | true / false | 主威胁是否在碰撞路径上（预计最近距离 < 等效半径） |

### 数值特征（9 项，哨兵值 99.0 表示"无有效值"）

| 特征 | 含义 | k-NN 归一化尺度 |
|---|---|---|
| `ttc_s` | 主威胁最近接近时刻（秒） | /3.0 |
| `miss_m` | 预计最近距离（米） | /5.0 |
| `lateral_dist_m` | 主威胁横向距离 | /10.0 |
| `lateral_speed_mps` | 主威胁横向速度 | /10.0 |
| `closing_speed_mps` | 相对接近速度 | /20.0 |
| `ego_speed_mps` | 自车速度 | /25.0 |
| `hj_risk` | 主威胁的 HJ 状态风险 [0,1] | /1.0 |
| `threat_dist_m` | 主威胁直线距离 | /30.0 |
| `scene_risk` | 场景组合风险 | （不进向量） |

### 风险分带公式（反射层）

每个目标：`combined = 0.6 × HJ风险 + 0.4 × TTC紧迫度`，其中
`紧迫度 = clamp(1 − ttc/3s, 0, 1)`（仅碰撞路径上的目标有紧迫度）；
后方且不在碰撞路径的目标 HJ 权重 ×0.3（HJ 网络只训练过前方障碍）。
场景风险取各目标最大值：**< 0.40 为 low，< 0.63 为 elevated，否则 critical**。

**提交条件**（何时真正执行库匹配）：critical 带，**或** elevated 带但
主威胁在碰撞路径上且 `braking_sufficient=false`（"刹不出去"本身就是提交点，
远距静态障碍的 HJ 评分偏低，靠这条兜住）。

## 八、中等保真动力学仿真器

`tools/simulator.py` —— 离线工厂的"物理真值"来源，也是在线仲裁的比较器。
确定性、纯计算、单次机动仿真约 0.3 ms（毫秒级是它能同时服务在线仲裁与
离线网格扫描热路径的前提）。相对早期"点质心 + 圆"模型，升级为**中等保真
解析动力学**，三个要素在保持毫秒级的同时贴近真实道路：

1. **矩形车身 OBB 碰撞**（`_obb_circle_gap`）：自车按 长×宽 矩形 + 航向建模
   （EGO_LENGTH 4.6 / EGO_WIDTH 1.9 m），目标按类型等效为圆；接触间距与
   **撞击面在车体坐标系判定并随航向旋转**。这对 T 形漂移是本质性的 ——
   漂移时车身横过来，来自侧向的目标命中的是**车尾**而非车侧。撞击面判据：
   在车体系比较纵/横归一化边距，|纵|主导→前/尾，|横|主导→侧。
2. **摩擦圆约束**（`_friction_clip`）：常规机动(0–4)的合加速度受
   `|a| ≤ μ·g`（μ=0.9）耦合限制 —— 真实车辆不能同时全力制动+全力转向；
   T 形漂移(5/6)是主动失稳滑移，轮胎已饱和，单列更高的滑移上限
   `DRIFT_ACCEL_CAP`。
3. **执行器动力学**（`_apply_actuator`）：加速度指令经**一阶滞后
   (ACTUATOR_TAU 0.12 s) + jerk 上限 (JERK_MAX 40 m/s³)** 逐步建立，不再
   瞬时理想执行。短 TTC 下真实体现"机动来不及"——这也是为什么 T 形漂移
   只在"碰撞不可避、但 ~1.3 s 甩尾转头还来得及把车尾对准"的窄带里最优。

4. **T 形漂移两段式**（`_accel_command` 的 5/6 分支）：①旋转段
   (t < ~0.9s) 纵向制动 + 侧向甩尾把航向转至 ±90°，建立大滑移角、把车尾
   甩向撞击点；②滑行段（车身已横过来、轮胎饱和）摩擦沿**当前速度方向**
   擦除动能直至停下，不再持续侧向加速。这样末态有界且物理合理（车停在
   横置姿态、只侧滑几米），而非无界飞出；碰撞发生的早期阶段几何不变，
   故严重度结论不受影响。

自车轨迹由 `_rollout_ego` 数值积分（加速度指令 → 执行器 → 摩擦圆 → 积分）。
机动剖面见第五节表格（AEB −8；变道 bang-bang 侧向 ±3 到 4 m 侧速归零；
变道+制动叠加纵向 −4；T 形漂移见上第 4 点两段式）。

- **周围目标同样运动**：障碍物携带速度（`velocity`，散落货物/失控拖车，缺省
  静态）；参与者按意图演化 —— Maintain 恒速 / Emergency Braking 沿速度方向
  −6 m/s² 至停 / Lane Change 横移 4 m 后侧速归零。
- **演化假设**：验证策略树分支时，主威胁按该分支声称的演化仿真
  （maintains 按观测 / yields 强制制动 / accelerates 速度 ×1.3）。
- **严重度 0~4**：无接触按最小间距分 0/1；有接触按相对速度定基础级
  （<3 → 2 / <8 → 3 / ≥8 → 4），再按撞击面修正：**尾碰 −1**（吸能结构）、
  **侧碰 +1**（结构弱/EV 电池）、**涉行人 +2**；越界视同撞护栏。
- **HJ 末态约束**（参考项）：机动末态 HJ 风险不得劣于"不干预"基线 + 0.05。

> **一个诚实的物理结论**：升级后**变道+制动（code 3/4）几乎从不单场景严格
> 最优**——它被纯制动或纯变道支配。这不是建模缺陷，而是真实结论；架构让
> 3/4 作为策略树的条件分支存在，不单列为演示场景。

### 8.1 仿真后端（可插拔）`tools/sim_backends/`

- **`analytic`（默认）**：包装上述 `simulator.py`，确定性、毫秒级，用于
  在线保守仲裁与离线网格扫描热路径。这是生产后端。
- **`carla`（可选桩）**：高保真离线交叉验证后端。CARLA 太重（GB 级 + GPU、
  单次仿真秒级），**绝不进在线/扫描热路径**，仅用于对解析后端标注的关键
  格点做抽样复核。默认不安装；`carla_backend.py` 顶部有完整接入步骤与坐标
  对齐说明。用法：`get_backend("carla", host=..., port=...)`。

```python

b = get_backend("analytic")            # 默认，realtime_safe=True
r = b.simulate_maneuver(snapshot, code, primary_id="truck1")
```

## 九、库文件格式规范

三个库都是 JSONL（每行一条 JSON），可 git diff、可人工审阅、可直接编辑。

### 9.1 规则库 `maca/library/rules.jsonl`

```jsonc
{
  "rule_id": "seed-tdrift-right",       // 唯一 id；seed-* 为手写领域种子
  "version": 1,                          // upsert 时自增
  "status": "active",                    // candidate | active | deprecated
  "priority": 10,                        // 同特异度平局时的优先级
  "guards": {                            // 守卫：特征语言的任意子集
    "approach_sector": ["front_left", "left"],   // 离散：成员列表
    "on_collision_course": [true],
    "ttc_s": {"max": 1.3},                        // 数值：闭区间（可单边）
    "lateral_speed_mps": {"min": 4.0},
    "lateral_dist_m": {"max": 10.0},
    "pedestrian_side": ["none", "left"]           // 右漂移要求右侧无行人
  },
  "policy": {                            // 载荷：应急策略树
    "branches": [
      {"condition_type": "primary_threat_maintains",
       "condition": "laterally crossing threat keeps closing",
       "code": 6, "target_speed_mps": 0.0,
       "rationale": "...", "confidence": 0.9},
      // ... yields / default 等分支
    ]
  },
  "rationale": "规则依据（人可读）",
  "provenance": {                        // 来源：工厂规则带完整场景族定义，
    "source": "domain_knowledge_seed"    //   可复实例化用于抗漂移复验
  },
  "stats": {"validated_n": 8, "pass_rate": 1.0, "validated_at": "..."}
}
```

匹配排序：**特异度（守卫条数）降序 > priority 降序 > pass_rate 降序** ——
更具体的规则压过更宽泛的规则；工厂归纳的细规则可以覆盖种子规则的子区域。

面向机器审阅，完整可解析。

### 9.6 库成长日志 `Results/factory/library_growth.log`（人类可读）

**这是"生成案例、判断、完善规则"的统一文本日志** —— 与 9.5 的机器可读
批次报告互补，一行一个事件、按时间顺序追加，可直接 `tail -f` 观察工厂
的工作过程。所有工厂批次（含 --mock）都会写入同一个文件，永久累积。

记录的事件类型：

| 标签 | 含义 | 关键内容 |
|---|---|---|
| `[批次开始]` / `[批次结束]` | 一次工厂批次的起止 | 库现状统计、报告路径 |
| `[场景族]` | 一个参数化场景族被扫描 | 网格规模、规律/模糊点数、未覆盖数、假设 |
| `[队列处理]` / `[队列跳过]` | 运行时反馈项 | 跳过时注明"已被谁覆盖" |
| `[案例生成]` | **新案例三行详情** | ① 决策机动码 ② **仿真判断**（结局 severity + 描述）③ **评估**（有效性 + 可迁移教训） |
| `[案例退役]` | 案例被冲突裁决否决 | 退役原因 |
| `[规则完善]` | 候选规则过晋升电池 | maintains 机动码、检查数、通过率、→ active/candidate |
| `[提案被拒]` | 归纳提案未过网格复核 | 守卫 + 复核问题（数值证据） |
| `[规则修订]` | 既有规则被收紧/废弃 | 动作 + 理由 |
| `[归纳跳过]` | 区域/格点已覆盖，省 token | 跳过原因 |
| `[规则复验]` | 批末抗漂移复验 | 检查数、通过率、→ 仍在役/废弃 |

一条 `[案例生成]` 的实例（真实批次）：

```
[案例生成] case-1f569081 @ right-crossing-...@ego_speed=5.0,...,ttc=0.4 来源=committee
           决策: code 0（Full emergency braking）
           判断: 仿真结局 severity 3/4 —— Collision with threat at t=0.05s, front impact ...
           评估: effective=True | 教训: When TTC is extremely short (≈0.4 s) and ...
```

实现见 `maca/factory/growth_log.py`，路径由 `config.GROWTH_LOG_PATH` 指定；
在 `factory.py` 的每个关键事件处接线写入。

## 十一、一次在线决策的逐步走读

以 `python -m maca.run --scenario test1` 为例（十字路口，卡车以 8 m/s 从左侧
横穿，几何标定为碰撞不可避免；行人在左前方）：

1. **加载**：`RuntimeExecutor` 启动时把 rules.jsonl / cases.jsonl 读进内存
   （规则对象列表 + 案例向量矩阵）。
2. **反射监视**（~1 ms）：对触发帧算每个目标的 HJ 风险与 TTC 紧迫度。
   卡车 combined=0.769 → **critical**；主威胁=truck1（在碰撞路径上）。
3. **特征化**（~0.3 ms）：sector=front_left、ped_side=left、ttc=0.74、
   lateral_speed=8.0、braking_sufficient=…（仿真 code 0 → 侧碰 4 级 → false）。
4. **规则匹配**（~0.01 ms）：`seed-tdrift-right` 六条守卫全中。
   **案例 k-NN**（~0.5 ms）：此前冲突裁决产生的案例相似度 1.0 命中。
5. **演化分类**（~0.1 ms）：对比预览帧（卡车在 (25,-16)）与触发帧（(10,-8)）：
   主威胁未变、仍在碰撞路径、速度未显著变化 → `primary_threat_maintains`。
6. **执行**：规则分支 code 6 与案例分支 code 6 一致 → 直接执行。
   总时延 ~1.5 ms，全程零 LLM。
7. **结局仿真**（不在关键路径）：code 6 → 车尾受撞、相对速度 13.5 m/s →
   severity 3（对比：制动/变道全是 4 级）。记录落盘 `Results/runs/`。

若第 6 步两库分歧（如 `cross_escape`：规则说 6、案例说 2），则 arbiter 把
两个机动各仿真一遍（code 6 → 尾碰 3 级；code 2 → 干净逃逸 0 级），执行案例
的 code 2；因该案例本身是冲突裁决产物，不再重复入队。

## 十二、一个工厂批次的逐步走读

以 `python -m maca.factory --batch 1 --mock` 为例：

1. **阶段 1 消化队列**：逐项先过覆盖门控 —— 例如一条 `cross_escape` 的冲突
   记录，检索发现已有裁决案例覆盖 → `SKIPPED (already covered by case:...)`，
   零 token 消耗。未覆盖的 gap 则交委员会：决策（读仿真严重度表做权衡）→
   安全校验层逐分支验证 → 反思写教训 → 案例入库。
2. **阶段 2 规律发现**：
   - 生成器给出场景族，如"左侧横穿卡车，扫 ego_speed×ttc×lateral_speed
     共 12 点"（mock 轮换预置族；真实模式由 gpt-5.5 读覆盖简报自行设计）；
   - 网格扫描：12 点 × 8 机动 = 96 次仿真（约 3 秒），标出 8 个规律点 /
     4 个模糊点；
   - 覆盖门控：8 个规律点若已被现役规则最优覆盖 → `consolidation SKIPPED`；
     若有"规则命中但次优"的点 → 作为反例送归纳器；
   - 归纳：LLM 输出带数值边界的规则提案 → 逐点网格复核（不符打回，≤2 轮）→
     晋升电池（来源网格 + 全部静态场景，≥90% 通过 → active）；
   - 模糊点：逐个查覆盖，未覆盖的（上限 3 个/族）交委员会成案例。
3. **批末抗漂移**：抽样 ≤5 条现役工厂规则，用其 provenance 里的场景族定义
   重新实例化、重新扫描、重新验证 —— 失效即废弃。
4. **报告落盘** `Results/factory/`，控制台打印逐项摘要。

## 十三、如何扩展

### 13.1 添加一个测试场景

在 `maca/scenario/scenarios/` 新建 `<name>.json`：

```jsonc
{
  "name": "my_scenario",
  "description": "一句话描述 + 期望的最优机动",
  "snapshots": [
    { "t": -1.0, "ego": {...}, "obstacles": [...], "participants": [...] },  // 预览帧
    { "t":  0.0, "ego": {...}, "obstacles": [...], "participants": [...] }   // 触发帧
  ],
  "execution_outcomes": {}    // 可留空：v2 结局以仿真器为准
}
```

要点：
- 坐标约定：自车在原点，x 前向，y 正方向为**自车右侧**；
- ego 可带 `road_boundary_left/right`（不写则无边界约束）；
- 障碍物可带 `velocity`（不写则静态）；参与者可带 `intention`
  （Maintain / Emergency Braking / Left|Right Lane Change）；
- 预览帧 = 触发帧按相对速度回推 1 秒（供演化分类参照）。

**先标定再定稿**（强烈建议）：写完后用仿真器验证最优机动是否符合预期：

```python
from maca.scenario.loader import load_scenario
from maca.runtime.monitor import assess_scene
from maca.tools import simulator
sc = load_scenario("my_scenario")
a = assess_scene(sc.trigger)
print("band:", a.band(), "primary:", a.primary_threat)   # 触发帧应为 critical
r = simulator.rank_maneuvers(sc.trigger, primary_id=a.primary_threat)
print("best:", r["best"], "margin:", r["margin"])
print({c: v["severity"] for c, v in r["results"].items()})
```

### 13.2 手写一条规则

在 `rules.jsonl` 追加一行（格式见 9.1 节）。手写规则建议：
守卫宁窄勿宽（宽了会在陌生区域抢答）；`priority` 给高些以在同特异度时
压过工厂规则；写清 `rationale`。加完直接生效（在线每次运行重新加载）。

### 13.3 添加新的场景模式（pattern）

在 `factory/instantiate.py` 增加一个 `_<pattern>` 构造函数并注册到
`instantiate()` 的分发表、`PATTERNS` 列表、`generator._REQUIRED_PARAMS`
与 `schemas.FAMILY_SCHEMA` 的枚举 —— 生成器即可开始设计该模式的实验。

### 13.4 添加新特征

在 `features.py` 计算并放入 discrete/numeric（如参与 k-NN 需加进
`NUMERIC_SPEC` 或 one-hot 段），再把特征名加入 `schemas.FEATURE_KEYS`
枚举 —— 规则引擎无需任何改动（守卫全可选），归纳器即可开始使用它。
注意：改动向量编码后，旧案例的向量与新查询不完全可比，建议让工厂
重新覆盖一遍（旧案例失配会自然走 gap→重建流程）。

### 13.5 调仿真器与阈值

全部集中在 `config.py`：机动加速度（`AEB_DECEL`、`TDRIFT_LAT_ACCEL`…）、
时域步长、严重度权重（`SIDE_IMPACT_PENALTY`…）、风险分带（`RISK_LOW`、
`RISK_TRIGGER`）、匹配阈值（`CASE_SIM_THRESHOLD`）、晋升门槛
（`RULE_PROMOTION_PASS_RATE`）等。改动仿真参数后建议重跑
`python -m maca.factory --batch 0 --source queue --mock` 与全场景在线回归，
并留意下一批次的抗漂移复验（规则可能因物理变化被废弃 —— 这是特性）。

### 已知可改进点（诚实标注）

1. `cross_escape` 中 T 形种子规则仍"带病服役"（每次被案例仲裁压过）——
   修订通道（consolidator 的 amendments）已打通，但尚未在真实批次里触发
   对种子规则的收紧。
2. **次威胁走廊占用**（故障注入发现）：以主威胁为键的种子规则本身不校验
   逃逸方向是否被**另一辆车**占据 —— 甩尾可能朝着已被占用的一侧发起
   （见注入场景 `gap_secondary_right_corridor`）。设计上的补救是
   "反例→带堵塞走廊模板的扫描→收紧守卫/覆盖案例"这条回路，在该批次跑出
   之前，在线保守仲裁是唯一兜底；更稳妥的做法是给种子规则加走廊感知守卫。
3. 工厂偶尔归纳出守卫过窄的规则（网格边缘退化区间）—— 无害，且正是
   规则生命周期与抗漂移复验存在的理由；可在归纳提示词中进一步抑制。
4. 演化分类当前基于两帧对比；接入实车/仿真平台的连续帧流后可做更细的
   意图估计。
5. 案例向量编码变更时的旧案例迁移目前依赖"失配→gap→重建"的被动流程，
   可加一个主动的重编码脚本。

## 十五、常见问题 FAQ

**Q：在线平面真的一个大模型调用都没有吗？**
A：是。`maca/runtime/` 不 import openai（可 grep 验证），断网、不配 key
都能跑：`OPENAI_API_KEY= python -m maca.run --scenario test1`。

**Q：为什么不把深思放在线上后台异步跑？**
A：v1 就是这么做的（预谋式深思）。实测深思 12~135 秒，而典型预谋窗口只有
1~2 秒 —— 永远赶不上。与其维护一条"永远迟到"的在线深思路径，不如全部
离线化，把在线简化为纯匹配（这也是用户拍板的方向）。

**Q：规则和案例都命中但结论不同，听谁的？**
A：都不直接听 —— 把两个候选机动各仿真一遍，执行预测伤害小的那个（~0.1 ms），
同时把冲突交离线裁决，败诉方被退役/修订，下次不再冲突。

**Q：大模型幻觉怎么防？**
A：三道闸：(1) 生成器不产出坐标，实例化确定性构造 + 参数护栏；
(2) 归纳的规则边界逐点回网格数值复核，不符打回；(3) 委员会的策略过
确定性安全校验层（逐分支仿真 + 硬规则 + HJ 约束），最多两轮否决后
强制替换为仿真最优机动。

**Q：`--mock` 和真实模式差在哪？**
A：管线完全相同，只是把 LLM 换成确定性替身（生成器轮换 3 个预置场景族、
归纳走无 LLM 基线、委员会决策直接采纳仿真最优）。用于零成本回归测试
与演示；真实模式的增量价值在生成器的实验创造性与委员会的权衡质量。

**Q：HJ 网络在哪里起作用？**
A：两处 —— 反射层风险信号（分带）与机动末态的安全约束参考（不得劣于
不干预基线）。它不做最终裁决；权重取自 HJ_Reachability/safe_value_params.pth（逐位可复现，`risk/hj_model.py`）。

**Q：怎么复现论文里的时延数字？**
A：跑任意场景加 `--json`，看 `timings` 字段；或看 `Results/runs/` 里的记录。
注意首跑含 torch 冷启动（~15 ms），热态为 1~2 ms。

## 附录 B：术语表

| 术语 | 含义 |
|---|---|
| 双平面 / 经验编译 | 离线 LLM 工厂产出库、在线只匹配库的总架构；"编译器/编译产物"类比 |
| 反射层（Reflex） | 在线平面的风险监视 + 分带逻辑，毫秒级、无 LLM |
| 风险分带 | low / elevated / critical 三带，由 HJ 风险与 TTC 紧迫度加权组合划分 |
| 提交条件（commit） | 真正执行库匹配的条件：critical 带，或"在碰撞路径上且刹不出去" |
| 应急策略树（policy tree） | 规则/案例的载荷：按 5 类演化条件分支的机动预案 |
| 演化分类 | 触发时刻对比预览帧/触发帧，把场景演化归入 5 类分支守卫之一 |
| 守卫（guards） | 规则的匹配条件：特征语言上的离散成员列表 + 数值区间 |
| 特异度（specificity） | 规则守卫条数；匹配排序第一键（更具体的规则优先） |
| 场景族（family） | 生成 Agent 设计的参数化实验模板（模式 + 参数范围 + 上下文） |
| 网格扫描（sweep） | 对场景族参数网格逐点仿真全部机动并标注最优的确定性实验 |
| 规律格点 / 模糊格点 | 某机动明显占优的点（归纳规则）/ 有害平局或涉行人权衡的点（交委员会） |
| margin | 网格点上最优机动对次优的严重度优势（级数）；≥1 判为规律格点 |
| 晋升电池（battery） | 候选规则转正前必须通过的回归检查集（来源网格 + 静态场景） |
| 抗漂移复验 | 每批次抽样重验现役规则（重实例化其来源场景族），失效即废弃 |
| 新颖性/覆盖门控 | "库未正确处理才花 LLM token"的前置检查 |
| 反例（counterexample） | 规则命中但严重度次优的格点，回喂归纳器用于修订 |
| 保守仲裁 | 规则/案例分歧时，两候选各仿真取预测伤害小者的确定性裁决 |
| 自愈 | 冲突离线裁决后败诉方被退役/修订，使同一分歧不再复现 |
| MRM | 最小风险机动（minimal-risk maneuver）：无覆盖时的 code 0 兜底 |
| TTC | Time-To-Collision 及其推广（最近接近时刻），紧迫度的来源 |
| HJ 可达性 | Hamilton–Jacobi 可达性：值网络给出的状态风险评估依据 |
| T 形漂移 | codes 5/6：旋转车身让车尾（吸能结构）承受不可避免的撞击 |
| 严重度（severity） | 仿真结局的 0~4 分级（无事/险情/轻微/中等/严重） |
| 蒸馏出口 | 委员会深思自动导出的微调样本流（慢教快的接口） |
| MockLLM / --mock | 全流水线的确定性离线替身，用于零成本回归与演示 |

## 附录 C：故障排查

| 现象 | 原因与处理 |
|---|---|
| 首次运行打印 torch `weights_only` Warning | torch 加载 HJ 权重的提示，无害，可忽略 |
| 首跑触发路径 ~15–20 ms 而非 1–2 ms | torch 首次推理的冷启动开销；热态即恢复，论文取热态值 |
| `Scenario 'xxx' not found` | 场景名需省略 `.json`；用 `--list` 查看全部可用名 |
| `queue.jsonl` 不存在 | 正常 —— 只有出现 gap/冲突才会创建；工厂 drain 后会删除 |
| 工厂报 `OPENAI_API_KEY is not set` | 在 `Code/.env` 配 key，或加 `--mock` 走离线替身 |
| 工厂真实模式连接超时/断连 | 网络/代理问题；SDK 自带重试，仍失败则稍后重跑该批次（批次幂等：库为追加式） |
| 结构化输出报 `Invalid schema ... required` | OpenAI strict 模式要求 object 属性全进 required 且禁自由 dict；参照 `schemas.py` 的条目数组写法 |
| 改了特征/向量编码后旧案例不再命中 | 预期行为：失配场景会走 gap→工厂→重建；也可手动清理旧案例 |
| 改了仿真参数后某条规则被复验废弃 | 预期行为（抗漂移）：物理变了规则就该重新验证；工厂会重新归纳 |
| 想看某次决策的完整依据 | 加 `--json`，或读 `Results/runs/<scenario>-<时间戳>.json`（含分项时延、命中详情、仲裁报告） |
| conda 环境不叫 `llm_control` | 任何满足 requirements.txt 的 Python ≥3.9 环境均可 |
